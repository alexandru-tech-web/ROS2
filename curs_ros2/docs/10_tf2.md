# Modulul 10 — TF2 (transformări între sisteme de coordonate)

## Ce înveți
- Ce este un **frame** (sistem de coordonate) și de ce un robot are mai multe.
- Ce este **arborele TF** și cum se leagă frame-urile unul de altul.
- Cum **publici** o transformare dinamică cu `TransformBroadcaster` și un mesaj `TransformStamped`.
- Cum **citești** o transformare cu `Buffer` + `TransformListener` și `lookup_transform`.
- De ce rotațiile se exprimă ca **quaternioni** și cum obții unul dintr-un yaw.
- Diferența dintre transformări **statice** și **dinamice**.

## Conceptul, pe scurt
Un **frame** este un sistem de coordonate: o origine și niște axe (x, y, z). Un robot real are zeci de frame-uri: unul pentru baza robotului, unul pentru fiecare roată, unul pentru cameră, unul pentru fiecare senzor, plus unul pentru "lume" (un punct fix din mediu).

**Problema:** camera vede un obstacol "la 2 metri în fața mea". Dar roțile, planificatorul de traiectorie și harta gândesc fiecare în **propriul** sistem de coordonate. Ca să folosești informația, trebuie să o **traduci** din frame-ul camerei în frame-ul roților, al bazei sau al lumii.

**Analogie — hărți suprapuse.** Imaginează-ți o hartă a orașului și, peste ea, o folie transparentă cu harta cartierului tău, lipită ușor înclinată și deplasată. Un punct desenat pe folie ("colțul străzii mele") are o poziție pe folie, dar **și** una pe harta orașului. TF2 este exact regula care spune "cu cât e deplasată și cu cât e rotită folia față de hartă", ca să poți converti orice punct dintr-o hartă în cealaltă. Fiecare frame este o astfel de folie; TF2 ține minte cum sunt aliniate între ele.

### Arborele TF
Frame-urile formează un **arbore**: fiecare frame (în afară de rădăcină) are exact **un părinte**. O transformare leagă un părinte de un copil (`world` → `robot`). Pentru a afla relația dintre două frame-uri oarecare, TF2 urcă și coboară pe ramuri și **compune** transformările de pe drum. De aici regulile:
- Un copil are **un singur** părinte (altfel arborele devine ambiguu).
- Nu sunt voie bucle (ar fi un graf, nu un arbore).

În acest modul construim un arbore minimal: `world` (rădăcina, fixă) → `robot` (copilul, care se mișcă).

## Codul-cheie

### Quaternion dintr-un yaw
În ROS, orientarea **nu** se stochează ca unghiuri Euler, ci ca **quaternion** (x, y, z, w). Quaternionii evită ambiguitățile și problema de "gimbal lock". Pentru o rotație doar în jurul axei verticale Z (un yaw), formula se simplifică:

```python
import math

def quaternion_din_yaw(yaw):
    # Rotatie pura pe Z -> componentele x si y sunt 0.
    qx = 0.0
    qy = 0.0
    qz = math.sin(yaw / 2.0)
    qw = math.cos(yaw / 2.0)
    return (qx, qy, qz, qw)
```

Observă `yaw / 2.0`: quaternionul folosește jumătate de unghi. Iar `qz**2 + qw**2 = sin² + cos² = 1`, deci quaternionul iese **normalizat** automat.

### Broadcaster (publică transformarea)
Robotul se plimbă pe un cerc și se rotește; publicăm transformarea la 20 Hz:

```python
from tf2_ros import TransformBroadcaster
from geometry_msgs.msg import TransformStamped

self.br = TransformBroadcaster(self)
# ... in callback-ul de timer:
t = TransformStamped()
t.header.stamp = self.get_clock().now().to_msg()  # ACUM
t.header.frame_id = 'world'                        # parintele
t.child_frame_id = 'robot'                         # copilul
t.transform.translation.x = math.cos(self.t)
t.transform.translation.y = math.sin(self.t)
t.transform.translation.z = 0.0
qx, qy, qz, qw = quaternion_din_yaw(self.t)
t.transform.rotation.x = qx
t.transform.rotation.y = qy
t.transform.rotation.z = qz
t.transform.rotation.w = qw
self.br.sendTransform(t)
self.t += 0.05
```

### Listener (citește transformarea)
Cu `Buffer` (memoria) și `TransformListener` (urechea care o umple), interogăm relația o dată pe secundă:

```python
from tf2_ros import Buffer, TransformListener, TransformException

self.buffer = Buffer()
self.listener = TransformListener(self.buffer, self)
# ... in callback-ul de timer:
try:
    trans = self.buffer.lookup_transform(
        'world', 'robot',
        rclpy.time.Time()   # timp 0 = cea mai RECENTA transformare
    )
except TransformException as e:
    self.get_logger().info(f'Inca astept transformarea: {e}')
    return

x = trans.transform.translation.x
y = trans.transform.translation.y
```

`rclpy.time.Time()` (timpul 0) este trucul important: cere **cea mai recentă** transformare disponibilă, în loc de una de la un moment fix. Evită erorile de "timp în viitor" sau "timp prea vechi".

## Static vs. dinamic
- **Dinamic** (ce facem aici): transformarea se schimbă în timp (robotul se mișcă), deci o **republici** mereu, cu un `TransformBroadcaster` și un timer. Fiecare mesaj are un `stamp` nou.
- **Static**: transformarea **nu** se schimbă niciodată (ex.: o cameră înșurubată rigid pe corpul robotului). O publici **o singură dată**, cu `StaticTransformBroadcaster`. TF2 o reține "pentru totdeauna", deci e mai eficient decât să o trimiți de 20 de ori pe secundă degeaba.

Pentru o transformare statică rapidă, fără cod, există un nod gata făcut:

```bash
# x y z yaw pitch roll frame_parinte frame_copil
ros2 run tf2_ros static_transform_publisher 0 0 0.2 0 0 0 robot senzor
```

Asta declară un frame `senzor` fix, la 0.2 m deasupra lui `robot`.

## Capcane frecvente
- **Timp greșit la lookup.** Dacă ceri o transformare la un moment care nu există încă în buffer (un timp din viitor sau dinaintea primului mesaj), `lookup_transform` aruncă `TransformException`. Cea mai sigură variantă pentru "cel mai recent" este `rclpy.time.Time()` (timpul 0). De aceea **prinde mereu** `TransformException` și reîncearcă.
- **`frame_id` inconsistente.** Listener-ul caută exact numele scrise de broadcaster. Dacă publici `world` → `robot` dar cauți `map` → `robot`, lookup-ul eșuează la nesfârșit. Nume identice, fără spații sau majuscule diferite.
- **Quaternion nenormalizat.** Un quaternion trebuie să aibă norma 1 (`x²+y²+z²+w² = 1`). Dacă pui valori la întâmplare (sau uiți `w`, care rămâne 0), orientarea e invalidă: rviz2 afișează ciudat, iar compunerile de transformări ies greșit. Formula din `quaternion_din_yaw` iese normalizată automat.
- **Părinte/copil inversate.** `header.frame_id` este **părintele**, `child_frame_id` este **copilul**. Dacă le inversezi, robotul "se mișcă" în direcția greșită și arborele iese pe dos.
- **Lipsește `stamp`.** Dacă uiți `t.header.stamp = self.get_clock().now().to_msg()`, mesajul are timp 0 și listener-ul nu mai poate decide ce transformare e "recentă".
- **Bufferul golit prea repede.** TF2 reține transformările doar pe o fereastră de timp (implicit ~10 s). Dacă broadcaster-ul s-a oprit demult, lookup-ul "celui mai recent" tot poate eșua pentru că memoria s-a golit.
